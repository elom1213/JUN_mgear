## 1. 문제 요약

광학식 모캡(Shogun/Motive)에서 나온 데이터는 **본 스켈레톤 + 거기 박힌 키프레임**이
전부다. 컨트롤러도, 컨스트레인트도 없다. 반면 우리가 가진 건 mGear가 만든 **빈 컨트롤
리그**다. 애니메이터가 손볼 수 있으려면 모캡 모션이 **컨트롤러 위에 키로 올라가야** 한다.

핵심 난점: 모캡 스켈레톤과 mGear 컨트롤 리그는 **스켈레톤 구조·비율·네이밍이 서로 다르다.**
그래서 본 키를 그대로 복사할 수 없고, **리타게팅(retargeting)** 이 필요하다.

## 2. 핵심 개념 — 왜 그냥 못 옮기나

> mGear 컨트롤러는 mGear 리그가 빌드 때 만든 **자기 디폼 조인트**를 구동한다([[06_원본_연결과_스키닝_노트]] 참조).
> 외부 모캡 본과는 아무 관계가 없다.

- 방향이 거꾸로다: 모캡은 "본 → (메시)" 흐름이지만, 우리는 "**컨트롤러 → 리그 → 조인트**" 흐름.
  즉 **매 프레임 컨트롤러를 모캡 포즈에 맞춰 구동**시킨 뒤, 그 결과를 컨트롤러에 **베이크**해야 한다.
- 비율 차이는 **HumanIK(HIK)** 가 해결한다. HIK는 "소스 캐릭터(모캡)"의 모션을
  "타깃 캐릭터(mGear 리그)"의 **비율에 맞게 자동 리타게팅**한다.
- **EPIC_mannequin / biped 템플릿은 HumanIK 친화적**이라 이 경로가 권장된다.
  mGear는 컨트롤 리그를 HIK에 연결해주는 **HumanIK Mapper** 도구를 제공한다.

## 3. (메인) 방법 A — HumanIK 리타게팅 + mGear HumanIK Mapper 베이크  ★권장

서로 다른 스켈레톤/비율을 HIK가 흡수해주고, mGear HumanIK Mapper가 리타게팅된 모션을
**컨트롤러(FK + IK)** 에 베이크해준다. biped/EPIC처럼 표준 인간형이면 이 경로가 정석.

1. **사전 준비**: mGear 리그를 빌드/오픈하고, 정리한 모캡 FBX/BVH를 **같은 씬**에 임포트한다.
   업축·스케일·fps를 리그와 일치시킨다(→ 5번 체크리스트).
2. **모캡 스켈레톤을 HumanIK Source로 캐릭터라이즈**한다.
   - Maya `Windows > Animation Editors > HumanIK` 패널에서 Character 정의 생성.
   - 모캡 조인트를 HIK 슬롯(Hips/Spine/Chest/Neck/Head, L·R UpperArm/Forearm/Hand,
     L·R UpLeg/Leg/Foot ...)에 매핑하고 **정의를 Lock**.
3. **mGear 컨트롤 리그를 HumanIK Mapper로 타깃 매핑**한다.
   - mGear 5.x: **`mGear` 메뉴 > Mocap > HumanIK Mapper** (정확 라벨은 8번 확인).
   - 표준 biped/EPIC은 컨트롤→HIK 이펙터 매핑이 **대부분 사전 정의**됨. 비표준 리그는
     **Custom Rig 매핑**(Bake To Custom Rig)으로 컨트롤을 직접 지정.
4. HIK 패널에서 **Source = 모캡 캐릭터, Target = mGear 리그**로 지정 → 타임라인을 스크럽해
   리그가 모캡을 따라오는지 확인한다.
5. HumanIK Mapper에서 **Add Sub IK → Activate Sub IK**.
   - Sub IK가 있어야 FK 체인뿐 아니라 **IK 핸드/풋 컨트롤까지** 키가 박힌다.
6. **Bake**(또는 mGear 5.x의 **Batch Bake to Timeline**)로 전체 프레임 범위를
   컨트롤러에 플롯한다.
7. **베이크 후 정리**: Euler 필터 적용, IK/FK·스페이스 스위치 상태 확인,
   모캡 스켈레톤/HIK 캐릭터를 삭제(또는 레퍼런스로 분리), 리그 편집 가능 여부 점검(→ 6번).

> ⚠️ 메뉴 경로·버튼명(**Add Sub IK / Activate Sub IK / Bake / Batch Bake to Timeline /
> Custom Rig**)은 mGear 버전마다 다르다 → 실제 Maya UI에서 확인(8번).
> ⚠️ 일부 버전(5.0.6)에서 Bake가 **RecursionError**로 실패한다(GitHub 이슈 #507).
> → **5.0.7+ 로 업데이트**하거나, 임시로 Maya 네이티브 HIK Bake 사용(단 네이티브는
> 선택된 스켈레톤 하나(IK 또는 FK)만 구워 FK+IK 동시 편의가 사라짐).

## 4. (폴백) 방법 B — 컨스트레인트 + 베이크 (HumanIK 없이)

단순·확실한 폴백. **언제 쓰나**: Bake 버그나 네임스페이스 문제로 캐릭터라이즈가 막힐 때,
스켈레톤이 충분히 비슷해 리타게팅이 불필요할 때, 또는 어느 컨트롤에 키를 박을지 **수동으로
완전히 통제**하고 싶을 때.

1. 스켈레톤 사전 준비(5번). 모캡 기준 포즈와 리그 빌드 포즈를 일치시킨다.
2. **부위별 IK/FK를 베이크 전에 결정**(둘을 동시에 구동하지 않는다).
   - FK 결과: 각 FK 컨트롤을 대응 모캡 본에 **orientConstraint**, 루트/Hips/COG 컨트롤에 **pointConstraint**.
   - IK 결과: IK 핸드/풋 컨트롤을 모캡 손목/발목(또는 그 위 로케이터)에 **parent/point+orientConstraint**, COG도 구동.
   - 폴벡터는 절차적으로 두거나 구동 로케이터로 제어.
3. 컨스트레인트는 **Maintain Offset(오프셋 유지)** 켜고 건다(컨트롤 기준 회전축/오프셋 보존).
4. **Edit > Bake Simulation** 으로 컨트롤러만 베이크(**디폼 스켈레톤은 베이크하지 않는다**).
5. **컨스트레인트 삭제** → 키는 컨트롤러에 남고 리그는 자유로워진다.
6. Euler 필터 적용. 필요하면 IK/FK 매치 패스를 돌린다.
   - mGear API: `mgear.core.anim_utils`의 `ikFkMatch_with_namespace`,
     `changeSpace_with_namespace`, `reset_all_keyable_attributes` 등.

## 5. 모캡 데이터 사전 준비 체크리스트 (Shogun / Motive)

- **업축(가장 중요)**: 템플릿이 *y_up* → Maya 씬을 **Y-up**으로 둔다.
  - **Vicon Shogun**: Shogun Post의 FBX 익스포트에서 **Up axis = Y** 로 설정(이게 Maya
    임포트/애니메이션 틀어짐을 막는다). Vicon 제공 `CleanupFBXImport.py`도 활용 가능.
  - **OptiTrack Motive**: 본래 Y-up이라 Maya와 맞지만, **FBX 익스포트에 축 변환 옵션이 없다**
    → 틀어지면 임포트 후 Maya에서 보정. (C3D보다 FBX 권장)
  - ⚠️ EPIC_mannequin에는 **Y-Up / Z-Up 변종**이 모두 있다. 모캡 업축과 다른 템플릿을
    섞지 말 것(섞으면 "작은/90도 회전된 레퍼런스 리그" 증상).
- **스케일/단위**: 모캡은 보통 cm/mm. Maya 작업단위를 **cm**로 맞춰 모캡 본이 리그와
  같은 키로 들어오게 한다(100배/0.01배 = 거대/초소형 레퍼런스 리그 증상).
- **싱글 루트**: HIK 캐릭터라이즈는 **깨끗한 단일 Hips/루트**를 요구한다 → 여분의
  reference/world/prefix 노드를 정리해 Hips가 최상위가 되게 한다.
- **레퍼런스 포즈**: HIK는 **T-pose**를 선호. 리그 빌드 포즈(A-pose 등)와 모캡 기준 포즈를
  맞추거나 HIK stance pose를 지정해 **상수 오프셋**이 베이크되지 않게 한다.
- **fps**: 임포트 **전에** 씬 fps를 캡쳐 레이트(30/60/120 등)에 맞추고, FBX 임포트에서도
  맞춰 키가 정수 프레임에 떨어지게 한다. 타임라인 범위 = 전체 클립.
- **네이밍**: Shogun/Motive 본 이름은 서로, 그리고 mGear와도 다르다 → HIK 슬롯이 이름을
  추상화해준다(메인을 HIK로 잡는 이유).

## 6. ⚠️ 주의점 / Gotchas

- **IK vs FK는 베이크 전에 결정.** Sub IK가 IK 키를 만든다. FK·IK를 **동시에 구동하지 말 것.**
  보통 다리는 IK(발 고정) + 팔은 샷에 따라. 상태 전환은 베이크 전 IK/FK 매치로 처리.
- **스페이스 스위치를 중립/알려진 상태로 리셋**하고 베이크(`reset_all_keyable_attributes`).
  블렌드 중간 상태에서 베이크하면 월드 결과가 안 맞는다.
- **컨트롤러에 베이크**(디폼 스켈레톤 X). 조인트에 구우면 리그가 이중 구동·편집 불가가 된다.
- **더블 트랜스폼**: Maintain Offset 미사용, 또는 베이크 후 모캡 소스를 안 지우면 모션이 이중 적용.
  베이크 후 컨스트레인트 삭제 + 모캡 소스 제거/디스커넥트.
- **Euler/짐벌**: 모캡 회전은 짐벌에 취약 → 베이크 후 **Graph Editor > Curves > Euler Filter**.
  필요하면 컨트롤 회전 순서(rotation order)를 베이크 전에 적절히 설정.
- **축/스케일 불일치 = "작은/Y-up으로 누운/90도 회전된 레퍼런스 리그"** 증상.
  캐릭터라이즈 **전에** 교정한다.
- **네임스페이스**: 리그가 네임스페이스로 레퍼런스되면 Characterize가 실패할 수 있다
  → 폴백(방법 B)을 쓰거나, 레퍼런스 전에 캐릭터라이즈.
- **리그 편집 가능성 점검**: 베이크 후 스크럽 → 컨트롤 하나 수정해 깔끔히 오버라이드되는지,
  IK/FK 스위치·스페이스 스위치가 정상인지 확인. **언베이크 백업**을 따로 저장.
