# 컨트롤러로 원본 메시·조인트 움직이기 (원본 연결 / 스키닝)

> [!info] 이 노트의 기원
> 2026-06-16 대화에서 출발 — "어깨·팔 등 부위별 컨트롤러를 만들었다. 이제 컨트롤러가
> 움직이면 **원본 메시와 원본 조인트도 따라 움직이게** 연결하고 싶다."
> 본인은 **이미 원본 스켈레톤이 있고 메시가 거기에 스킨된 상태**라, 그 원본 조인트를
> mGear 컨트롤러가 구동하게 하는 것이 목표. 기획 단계 → `기획_원본_연결과_스키닝`
> 전체 질문 흐름 → [[질문_기록]]

> 상위: [[00_MOC_mgear]] · 직전 단계: [[05_컴포넌트_연결_노트]]

## 1. 문제 요약

컨트롤러를 만들고 [[05_컴포넌트_연결_노트]]대로 서로 연결까지 했는데도, 컨트롤러를 움직이면
**원본 메시·원본 조인트는 그대로** 있다.

이유: **컨트롤러는 mGear가 빌드 때 만든 "자기 조인트"만 움직인다.** 사용자의 원본
스켈레톤/메시와는 아직 아무 관계가 없다. 그래서 둘을 이어주는 **마지막 연결 단계**가 필요하다.

## 2. 핵심 개념

> mGear 리그는 빌드 시 **자체 디폼 조인트(deformation joints)** 를 만들고, 이 조인트는
> 이미 컨트롤러에 묶여 있다. 우리가 할 일은 **원본 조인트가 이 mGear 디폼 조인트를
> 따라가게** 잇는 것.

- 빌드 후 아웃라이너의 `rig_sets_grp > **rig_deformers_grp**` 셀렉션 셋에
  **스킨/연결 대상이 되는 mGear 디폼 조인트가 전부 모여** 있다.
- 우리 흐름(원본 스켈레톤 보존):
  **컨트롤러 → mGear 디폼 조인트 → (연결)원본 조인트 → (기존 스킨)원본 메시**
- 원본 메시의 skinCluster는 **건드리지 않는다.** 원본 조인트만 mGear를 따라가게 하면
  메시는 자동으로 변형된다.

## 3. (메인) 시나리오 B — 기존 원본 스켈레톤 연결

전제: **원본 조인트에 메시가 이미 스킨돼 있다.** 이 스킨을 유지한 채 원본 조인트를
mGear가 구동하게 만든다.

### 방법 B1 — mGear 네이티브 "기존 조인트에 연결" (권장)

mGear는 생성한 디폼 조인트를 **씬에 이미 있는 조인트에 연결**하는 기능을 제공한다.
(리그를 갱신해도 스킨 웨이트를 잃지 않게 하는 용도와 동일한 메커니즘 — **매트릭스 컨스트레인트**)

1. 각 컴포넌트의 **Settings → Joint Settings(조인트 설정)** 를 연다.
2. 해당 컴포넌트가 만드는 조인트를 **씬의 기존 조인트에 연결**하도록 지정한다.
   - **연결할 원본 조인트 이름** 입력
   - **point / orient / scale** 각각을 적용할지 **0/1 플래그**로 지정
3. 빌드(또는 재연결)하면 mGear가 그 원본 조인트를 컨트롤러에 묶는다 → 원본 조인트가
   컨트롤러를 따라 움직이고, 거기 스킨된 메시도 함께 변형된다.

> ⚠️ Joint Settings의 정확한 항목 라벨/위치는 mGear 버전마다 다를 수 있다.
> 실제 UI에서 "connect to existing joint / use existing joint" 류 옵션을 찾아 확인할 것
> (아래 9번 미확정 항목).

### 방법 B2 — 수동 컨스트레인트 (단순·확실한 폴백)

mGear 네이티브 옵션을 쓰기 어렵거나 동작을 직접 통제하고 싶을 때:

1. 빌드 후 `rig_deformers_grp`에서 mGear 디폼 조인트들을 확인한다.
2. **원본 조인트 ↔ mGear 디폼 조인트**를 부위별로 1:1 매핑(이름·위치 대조).
3. 각 **원본 조인트**(driven)에 대응 **mGear 디폼 조인트**(driver)로
   **parentConstraint**(이동·회전). 스케일도 필요하면 **scaleConstraint** 추가.
   - 방향 주의: driver = mGear 조인트, driven = 원본 조인트.
4. 원본 메시 스킨은 그대로 → 원본 조인트가 mGear를 따라가며 메시 변형.
5. 컨트롤러를 움직여 의도대로 따라오는지 확인.

### 공통 주의점
- **조인트 오리엔트/회전축 불일치**: 원본과 mGear 조인트의 축이 다르면 회전이 틀어진다.
- **스케일**: 스케일까지 따라가야 하면 scale 연결을 빼먹지 말 것(점프/찌그러짐 방지).
- **더블 트랜스폼**: 원본 조인트가 다른 부모 밑에 있으면 변형이 이중 적용될 수 있다 — 계층 확인.
- **이름 매칭**: B1은 조인트 이름에 의존하므로 원본 스켈레톤 네이밍을 미리 정리하면 편하다.

## 4. (보조) 시나리오 A — 새로 스킨 바인드

원본 스켈레톤을 **유지할 필요가 없다면** 더 단순하다:
`rig_deformers_grp`의 mGear 디폼 조인트에 원본 **메시를 직접 `skinCluster`로 바인드**하면 끝.
(이 경우 원본 조인트는 버리고 mGear 디폼 조인트를 디폼 스켈레톤으로 채택.)
→ 우리 목표(원본 조인트 보존)와는 다르므로 비교용으로만 기억.

## 5. mGear 스킨 도구 (Skinning / Skin and Weights)

스킨 웨이트를 저장/복원/이동할 때 쓰는 mGear 도구:

| 도구 | 용도 |
|------|------|
| **Export / Import Skin** | 스킨 웨이트를 파일로 저장·로드 |
| **Copy Skin** | 한 메시의 웨이트를 다른 메시로 복사 |

스킨 파일 포맷:
- **`.gSkin`** — 바이너리(빠름)
- **`.jSkin`** — ASCII/텍스트 → 조인트 이름 오류 등을 **텍스트 에디터로 직접 수정 가능**
- **`.jSkinPack`** — 여러 `.jSkin`을 묶는 팩(지오 단위 관리)

> **Data-centric rigging**: mGear는 스킨/블렌드셰이프/모델/리그를 **각각 별도 파일**로 두고
> 빌드 때 합치는 워크플로우를 권장한다. Shifter는 빌드 시 미리 저장한 스킨(`.gSkin` 경로)을
> **자동 로드**할 수 있어, 리그를 다시 빌드해도 스킨을 잃지 않는다.
