# 🔁 IK/FK 매치 · 범위 베이크 (FK↔IK 모션 전환)

> [!info] 이 노트의 기원
> 2026-06-17 대화에서 출발 — "mGear로 세팅한 **biped 팔·다리 컨트롤러**에서,
> **FK 컨트롤러의 움직임을 간직한 채 IK 컨트롤러에 키를 베이크**하고 싶다(IK가 FK를
> 똑같이 따라함). 그리고 **그 반대(IK→FK)** 로도 베이크가 되는지 알고 싶다."
> 전체 질문 흐름 → [[질문_기록]]

> 상위: [[00_MOC_mgear]] · 관련: [[03_팔다리_IK_컴포넌트_레퍼런스]] · [[07_모션캡쳐_베이크_노트]]

> [!note] 적용 범위
> 이 노트는 **2족보행(biped) 캐릭터의 팔·다리 부위 컨트롤러**를 대상으로 한다.
> 즉 IK/FK가 내장된 **`arm_2jnt_01`**(팔), **`leg_2jnt_01` / `leg_3jnt_01`**(다리)
> 컴포넌트와 그 스위치를 담는 **`control_01`(Channel Host)**. 척추·손가락 등
> IK/FK 체인이 없는 부위는 대상이 아니다. (컴포넌트 설명 → [[03_팔다리_IK_컴포넌트_레퍼런스]])

## 1. 무엇을 하려는가

팔·다리 컴포넌트(`arm_2jnt_01` 등)는 **FK 컨트롤 체인**과 **IK 컨트롤(핸드/풋 + 폴벡터)**
**둘 다**를 갖고, `control_01`의 **blend 스위치 어트리뷰트**로 어느 쪽이 디폼을 구동할지 정한다
(→ [[03_팔다리_IK_컴포넌트_레퍼런스]]).

- 원하는 것: **FK로 만든 애니메이션을 IK 컨트롤러로 그대로 옮겨**, 포즈를 유지한 채
  IK 컨트롤에 키가 박히게 한다(=IK가 FK를 똑같이 따라감). **반대 방향(IK→FK)도 동일하게** 가능.
- 왜 단순 스위치로는 안 되나: `control_01`의 blend attr을 0↔1로 **토글만** 하면 **현재 프레임의
  포즈가 튄다**(반대 시스템엔 키가 없으니까). 구간 전체를 자연스럽게 옮기려면 **매치 + 범위 베이크**가 필요.

## 2. mGear가 제공하는 도구 — IK/FK Match & Transfer

mGear는 이 작업을 위한 전용 도구를 제공한다. 두 가지 층위가 있다.

| 도구 | 범위 | 용도 |
|------|------|------|
| **IK/FK Match** | **현재 프레임 1개** | 지금 포즈를 유지하며 IK↔FK 전환(포즈 안 튐). 키는 자동으로(관련 컨트롤에 이미 키가 있으면) 박힌다. |
| **IK/FK Transfer / Range Switch (Bake)** | **프레임 범위** | 지정한 구간 전체를 반대 시스템으로 **베이크**. 본 노트의 메인. |

- **호출 방법**:
  - 대상 **팔/다리 컨트롤을 선택 → 우클릭(mGear 컨텍스트 메뉴)** 에서 IK/FK Match / Transfer 항목.
  - 또는 mGear **Anim Picker**(synoptic)의 해당 팔/다리 IK/FK 버튼.
  - 좌/우 팔·다리는 **각각 따로** 수행한다(컴포넌트 단위 동작).
- **내부 API**: `mgear.core.anim_utils`
  - `ikFkMatch` / `ikFkMatch_with_namespace` — 현재 프레임 매치.
  - **`IkFkTransfer`** 클래스 — 범위 베이크(UI: `showUI()`, 코드: `execute()`).
- **양방향 지원**: FK→IK, IK→FK **둘 다** 같은 도구에서 방향만 바꿔 수행한다.

## 3. FK → IK 베이크 (IK가 FK를 똑같이 따라감)  ★사용자 요청 메인

1. **타임라인 범위**를 옮기려는 키가 **모두 포함**되도록 설정한다(예: 1–120).
2. 대상 **팔(또는 다리) 컨트롤 하나를 선택** → 우클릭 → IK/FK **Transfer**(범위) 다이얼로그를 연다.
3. 다이얼로그에서:
   - **방향 = FK → IK** (FK 모션을 IK로),
   - **프레임 범위**(start/end),
   - **베이크 방식**: **매 프레임**(every frame, 정확하게 일치) 또는 **키프레임만**(onlyKeyframes, 이후 편집 쉬움).
4. 실행 → IK **핸드/풋 컨트롤 + 폴벡터**에 키가 박히고, `control_01`의 **blend 스위치가 IK 쪽**으로 설정된다.
5. 스크럽해 **IK 컨트롤이 원래 FK 모션과 똑같이** 움직이는지 확인. 필요하면 **Euler 필터**(아래 6장).

> 결과적으로 FK 애니는 그대로 보존되고(원하면 백업 보관), 그 모션이 **IK 컨트롤 위에 키로** 올라간다.
> 이제 IK 핸들로 발/손 고정·핀 같은 IK 특유의 편집을 이어갈 수 있다.

## 4. IK → FK 베이크 (역방향)

같은 Transfer 다이얼로그에서 **방향만 IK → FK**로 바꿔 동일하게 실행한다.

- 결과: **FK 체인 각 컨트롤**(어깨/팔꿈치/손목 등)에 회전 키가 박히고 blend 스위치가 FK 쪽으로.
- 용도: IK로 잡은 모션을 **관절별 FK로 세밀 조정**하거나, FK 기반 파이프라인/엔진으로 넘길 때.

즉 **FK↔IK 양방향 모두** 한 도구로 왕복 베이크가 된다.

## 5. 스크립트 / 배치 (UI 없이)

여러 캐릭터·네임스페이스를 일괄 처리하거나 커스텀 스텝에 넣을 때는 `anim_utils`를 직접 호출한다.

```python
from mgear.core import anim_utils

# 현재 프레임 매치(예시 — 정확한 인자명은 아래 7장에서 확인)
anim_utils.ikFkMatch_with_namespace(namespace, ikfk_attr, ui_host, fks, ik, upv, ...)

# 범위 베이크: IkFkTransfer 사용 (start/end 프레임, onlyKeyframes 등)
anim_utils.IkFkTransfer.execute(...)
```

- 핵심 파라미터(개념): **컴포넌트/UI 호스트(`control_01`)**, **ikfk blend 어트리뷰트명**,
  **FK 컨트롤 리스트 / IK 컨트롤 / 업벡터(폴벡터) 컨트롤**, **start/end 프레임**, **onlyKeyframes 여부**.
- 모캡 워크플로우에서 FK 결과를 IK로 정밀화할 때도 같은 API를 쓴다(→ [[07_모션캡쳐_베이크_노트]] 4·6장).

## 6. ⚠️ 주의 / Gotchas

- **베이크 전 범위·키 확인**: 타임라인 범위 밖이나 키 없는 구간은 옮겨지지 않는다. 클립 전체를 포함시킬 것.
- **베이크 후 Euler 필터**: 회전이 튀면 **Graph Editor > Curves > Euler Filter**. 필요 시 베이크 전에 rotation order 정리.
- **Anim Layer 위에서는 매치가 어긋날 수 있음**(포럼 보고 버그). 가능하면 **Base 애니 레이어**에서 수행하거나 레이어 머지 후 진행.
- **컨트롤에만 베이크**, 디폼 조인트엔 굽지 않는다(→ [[07_모션캡쳐_베이크_노트]]와 동일 원칙). 조인트에 구우면 이중 구동.
- **매 프레임 vs 키만**: 매 프레임 = 원본과 정확히 일치(무겁고 곡선 빽빽), 키만 = 가볍고 편집 쉬움(미세하게 다를 수 있음). 목적에 맞게 선택.
- **좌/우, 팔/다리 각각** 수행해야 한다(한 번에 한 컴포넌트).
